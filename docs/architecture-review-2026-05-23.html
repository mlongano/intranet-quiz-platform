Problem

Three route handlers independently load the same snapshot, build the same qbank_map, iterate the same score entries, and issue individual UPDATEs. Each is a shallow module — the interface (parse request → load snapshot → iterate → update) nearly matches the implementation line-for-line.

Solution

Extract a transform_scores(session_id, teacher_id, fn) module. It loads the snapshot once, iterates score entries once, applies the caller's transformation callback to each entry's answers, and batch‑updates. The three route handlers become thin adapters that pass different callbacks.

Wins

  • locality: snapshot loading, qbank_map building, and iteration live in one module
  • leverage: one interface serves three callers; adding a fourth score mutation costs one callback
  • interface shrinks: route handlers drop ~60 lines of duplicated scaffolding each
  • test surface: one module to test with fake transformation callbacks

Problem

The JOIN query that loads a session's snapshot content, parses its JSONB, extracts questions, and builds a {str(id) → q} dictionary is repeated verbatim at 4 sites. Deleting each copy would shift the complexity to the remaining 3 — the deletion test confirms this is a pass‑through, not a module earning its keep.

Solution

Create a get_qbank_for_session(conn, session_id) → dict module in services/. Callers pass their open connection and receive a qbank_map. The JOIN query, JSONB handling, and id‑to‑string coercion concentrate in one place.

Wins

  • locality: change the query, the JSONB shape, or the indexing key → fix once
  • leverage: 4 callers, 1 interface, ~15 lines each evaporate
  • test surface: test the query and qbank construction independently of route handlers
  • enables candidate #1 (transform_scores would use this module)

Close the quiz‑submission transaction gap

Strong ✅ Done

routes/quiz.py:62–82,113–131 · services/quiz_session.py (find_plan_by_quiz_id, save_answer, submit_plan)

Before — two connections, no lock across ownership check

sequenceDiagram
    actor R as route handler
    participant P as find_plan_by_quiz_id
    participant D as DB conn 1
    participant S as save_answer
    participant D2 as DB conn 2

    R->>P: find_plan quiz_id

    D-->>P: plan data returned
    P-->>R: plan_data
    Note over R: validates student_id matches
but no lock held here
    R->>S: save_answer quiz_id, answer
    S->>D2: SELECT FOR UPDATE
    Note over D2: locks row too late

After — single transaction, ownership inside lock

sequenceDiagram
    actor R as route handler
    participant Q as save_answer
    participant D as DB single txn

    R->>Q: save_answer quiz_id, answer, student_id
    Q->>D: BEGIN
    Q->>D: SELECT FOR UPDATE
    D-->>Q: plan_data
    Q->>Q: assert student_id matches
    Q->>Q: modify progression
    Q->>Q: UPDATE quiz_plans
    Q->>D: COMMIT
    Q-->>R: result

Problem

The route handler validates plan ownership in one connection, then calls save_answer in another. Between these two connections, the plan could be deleted or reassigned. The correctness invariant — "only the owning student can answer" — is spread across two modules with no shared transaction. The FOR UPDATE in save_answer locks the row too late.

Solution

Push student_id into save_answer and submit_plan as a parameter. The service module opens one transaction, SELECTs FOR UPDATE, asserts ownership inside the lock, then proceeds. Remove the separate find_plan_by_quiz_id call from the route handlers — it becomes an internal detail of the service.

Wins

  • locality: the "ownership → mutation" invariant lives in one transaction, one module
  • leverage: route handlers shrink — no manual find_plan + check + call pattern
  • correctness: FOR UPDATE protects the full read‑check‑write cycle
  • test surface: save_answer can be tested with invalid student_id → FORBIDDEN in one call

Route‑service seam in teacher.py is blurry

After — clean seam

Problem

teacher.py is 781 lines. It imports from db, db.queries, services.*, email_service, and auth — sitting at 5 architectural layers. About 40% of its body is business logic (DB queries, JSON parsing, answer manipulation) that belongs behind the service seam. The route‑to‑service seam exists (services/) but it's inconsistently used.

Solution

Move ownership checks into a Flask before_request or param‑level decorator (session_id → verify belongs to g.current_user['sub']). Move inline DB queries for session/scores into service functions. Candidate #1 and #2 already pull most of the weight — this is the cleanup step after they land.

Wins

  • locality: business logic concentrates in services/; route file drops to ~300 lines
  • leverage: new teacher endpoints follow a single pattern — thin handler + service call
  • test surface: services testable without Flask; routes testable with mocked services
  • readability: one file to understand HTTP concerns, another for domain logic

Problem

The pattern x if isinstance(x, list) else json.loads(x or '[]') appears at ~9 sites across 3 files. It's a shallow pass‑through — deleting one copy shifts the work to the remaining 8. The _parse_json_field helper exists in quiz_session.py but isn't used by other modules.

Solution

Move _parse_json_field (and its two narrower variants: ensure_list, ensure_dict) from quiz_session.py to utils.py. The module already contains pure string/formatting helpers — adding JSON deserialisation guards fits. All callers import from one place.

Wins

  • locality: JSON deserialisation logic in one module, not 9 scattered sites
  • leverage: ~9 callers share 3 small functions
  • deletion test: remove the pattern from one site → complexity reappears at 8 others

Top recommendation

Candidate #1 — Collapse the score‑review / recalculate / regrade trio. It delivers the highest leverage (3 route handlers → 1 module + 3 thin callbacks) and the clearest test surface. Candidate #2 (snapshot loading) is a natural prerequisite — tackle it first as a 20‑minute extraction, then use it inside Candidate #1. Candidate #3 (transaction gap) is a correctness bug that should be fixed regardless of deepening — it's not speculative.
